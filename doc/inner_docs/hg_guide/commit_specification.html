代码提交规范

我们使用hg（mercurial）作为代码管理工具，并使用trac来进行 版本管理。在代码的提交过程中，我们需要遵守必要的提交规范，来保证 产品源码的可追溯性，方便我们对于源码的管理。

在阅读本规范之前，我们假设你已经了解hg和trac的基本功能和命名， 并安装了相关的hg客户端工具。

1 基本规范

  1. commit之前需要完成必要的自测，所有代码提交都需要保证编译通过，并不影响其他人员的正常研发
  2. commit需要和trac中的ticket关联。使用 "Refs #1123 ..." 或者 "Fixes #1123 ..."这样的语法， 1123为trac中的ticket号，trac通过 # 关联到具体的ticket
  3. commit之前最好通过hg pull -u拉下远程服务器上的代码并更新，这样可以有效的避免merge的次数。 如果已有本地的commit，则需要通过hg merge做合并，保证当前的代码分支上只有一个head
  4. 在完成上述操作之后，可以通过hg push将本地提交同步到代码服务器。除非你清楚自己在干什么，否则不要使用push -f

2 版本之间的合并规范

在产品研发中，同一module（例如ewp、ert-iOS）不可避免的会出现多个版本分支（例如4.3和5.2），而当我们需要在 不同的分支上提交同样的代码修改时，我们需要通过正确的hg命令和相关规范来保证分支之间的关联性。

我们在开发管理规范中将需要在多个分支上提交的代码修改分为两类：1）从低版本合并到高版本； 2）从高版本合并到低版本。 在日常工作中，我们面临的绝大部分是第一种情况

2.1 从低版本合并到高版本（或主干）

当低版本中暴露出的缺陷（bug）通常在高版本中也存在，此时我们需要将bug fix的代码提交合并到高版本中。此时，我们需要注意：

  1. 研发人员在本地需要分别clone下不同分支的源码，并首先在低版本（例如4.3）上做出bug fix的commit
  2. 确认需要合并的高版本分支（例如5.2），通过hg pull从本地的4.3代码分支中将4.3上的修改集拉到5.2分支（在5.2/ewp 目录下执行 hg pull path/to/4.3/ewp ,然后通过hg merge合并这些修改集，其中需要解决存在冲突，并通过hg backout放弃不需要的commit
  3. 合并之后在5.2/ewp分支上完成编译和自测，然后在4.3及5.2上分别通过hg push同步到远程服务器
  4. 在低版本（4.3）上提交不需要合并的代码修改时需要在提交注释中明确标识出来，例如 "Refs #1123 NoNeedToMerge ..."
  5. 当需要在多个版本上做合并时，采取渐进合并的方式，例如在5.1上的代码修改，需要首先merge到5.2，再从5.2 merge 到5.3

2.2 从高版本合并到低版本

由于高版本通常包含了较多低版本不需要的功能开发，所以我们往往不能够直接通过hg pull和merge来完成低版本所需的修改合并。
