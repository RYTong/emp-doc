2 引言

2.1 目的

本文旨在说明离线协议安全优化的整体设计，使得详细设计人员和开发人员理解需要实现的具体功能。

2.2 背景

目前，EMP离线协议规定的资源下载流程中，并未定义加解密和校验流程，离线资源在下载过程中使用明文传输，这使得含有用户信息的资源存在被窃取和篡改的风险。

3 总体设计

3.1 需求规定

3.1.1 软件功能说明

本次优化我们需要满足以下需求：

  1. 支持在服务端配置离线资源安全传输的方式：明文传输、EMP信道明文传输、EMP信道密文传输、HTTPS/SSL。
  2. 服务端根据配置在描述文件中增加传输方式字段，客户端根据该字段标识的传输方式下载资源。不符合规定的下载请求将被拒绝。
  3. 修改现有HTTP和TCP下载接口，使其支持四种传输方式。
  4. 修改服务端存储离线资源的路径，使资源不会被直接下载。
  5. 兼容旧版本客户端

4 接口设计

4.1 内部接口修改

4.1.1 HTTP接口

客户端在发送下载请求时，文件路径不再与Host拼接成URL。改为使用HTTP Post方法，将文件路径作为path参数传给服务端，此参数须放在Post Body中。

4.1.2 TCP接口

由于客户端请求时需要上送传输方式、签名、序列号、session id和文件路径等数据，因此需要修改原离线协议定义的客户端请求格式。服务端响应格式不变。修改后的请求格式如下。

客户端请求格式为EMPTCPVersion(2 byte) + RequestMessages，其中RequestMessages由若干消息组成，每个消息格式为Type(1 byte) + Length(4 bytes) + Content(Length bytes)组成。每个字段意义如下：

  • EMPTCPVersion第一字节表示大版本，第二字节表示小版本。
  • Type表示消息类型。0为session id，1为签名，2为序列号，3为文件路径。所有传输方式都需要上送文件路径。
  • Length表示消息的长度。
  • Content为消息内容。文件路径与原协议保持一致，使用JSON格式表示。 使用EMP信道时只对Content内容签名和加密。服务端在TCP下载响应中增加错误码3，表示安全验证失败。
  • 使用EMP信道密文传输时，对文件路径消息的Content进行加密。使用EMP信道传输时，对序列号消息的Content+文件路径消息的Content（密文传输时为加密后的内容）进行签名。

服务器响应格式为EMPTCPVersion(2 byte) + ResponseMessages，其中ResponseMessages由若干消息组成，格式与RequestMessages相同。Type类型包括0、1和2，其中0为资源内容，1为签名（使用EMP信道传输时会包括此消息），2为错误信息，服务端程序异常时只会包含错误信息。Type为2时，Content表示错误代码：为0时表示未知错误，为1时表示无法读取指定资源文件，为2时表示无法解析客户端上送的路径，为3时表示安全验证失败。

4.1.3 resource_hash接口

服务端会在resource_hash的响应JSON中增加st字段，表示安全传输的方式，客户端根据这个字段发送下载请求。st为0表示明文传输，为1表示EMP信道明文传输，为2表示EMP信道密文传输，为3表示HTTPS/SSL传输。

resource_hash接口新增版本号参数，传值为2.0（与信道合并的接口传res_ver参数，单独的接口传version参数）。resource_update接口版本号参数新增值2.0。
