4 配置

offline配置中增加encrypt_type和url配置项。EWP启动时，须根据这两项配置动态注册controller来生成HTTP下载接口映射。

{offline, [
    ...
    %% 0表示明文传输，默认值
    %% 1表示EMP信道明文传输，
    %% 2表示EMP信道密文传输，
    %% 3表示HTTPS/SSL传输。
    {encrypt_type, 0},
    %% http下载接口URL，默认为"ota/resource"。
    {url, "ota/resource"}
    ...
]}

5 资源存储路径

原来存储路径为APP/public/www/resources，且映射到ewp/public/www/APP/resources，该路径可以在外网环境被访问。为避免资源在外网被直接下载，我们需要修改资源存储路径，将其修改为APP/public/resources。

6 HTTP下载接口

客户端需要将文件路径作为path参数，放在Post Body中发送下载请求。出于兼容性考虑， 当APP/resources接口没有获取到path参数时，此时为旧客户端的下载请求，服务端需要根据URL解析下载路径。如URL为Host/ota/resource/iphone/320*480/txt/foo.txt时，下载路径为iphone/320*480/txt/foo.txt。服务端根据path参数或解析的相对路径到APP/public/resources查找指定文件。如查找不到文件，返回404状态码。

接口规范如下：

  • 接口名称：ota/resource
  • 接口参数：path
  • 接口返回：成功时返回200状态码及文件内容。失败时返回404状态码及错误信息。
  • 接口说明：path值为更新描述中的下载路径。接口名称会根据配置改变，客户端可从ota/resource_update接口返回的服务端地址获取接口名称。

7 TCP下载接口

接口规范如下：

  • 接口输入：EMPTCPVersion(2 byte) + RequestMessages，其中RequestMessages由若干消息组成，每个消息格式为Type(1 byte) + Length(4 bytes) + Content(Length bytes)组成。
  • 接口输出：EMPTCPVersion(1 byte) + ResponseMessages，其中ResponseMessages由若干消息组成，每个消息格式同输入。
  • 接口说明：
    • EMPTCPVersion固定为<<2,0>>。
    • Type表示消息类型。1为session id，2为签名，3为序列号，4为输入或输出内容。在RequestMessages中，4类型消息为文件路径。在ResponseMessages中，4类型消息为文件内容。
    • Length表示消息的长度。
    • Content为消息内容。文件路径与原协议保持一致，使用JSON格式表示。使用EMP信道时只对Type为4的消息Content内容做签名和加密。
    • 如果出现错误，在ResponseMessages会包涵Type为0的消息。此时，Content为错误代码：为0时表示未知错误，为1时表示无法读取指定资源文件，为2时表示无法解析客户端上送的路径，为3时表示安全验证失败。
    • 在RequestMessages中，必须包含4类型消息。1、2和3类型消息在使用EMP信道时使用。
    • 正常流程中，ResponseMessages必须包含Type为4的消息。使用EMP信道时，还会有Type为2的消息。异常流程中，0类型消息代替4类型消息返回错误信息，且无其他类型消息。
    • 为兼容旧版本，服务端须保留EMPTCPVersion为1时的处理逻辑。

8 资源描述更新接口

请求参数中，新增2.0版本号。

9 resource_hash接口

新增版本号参数，传值为2.0。未传时，视为1版本。

服务端会在resource_hash的响应JSON中增加st字段，表示安全传输的方式，客户端根据这个字段发送下载请求。st为0表示明文传输，为1表示EMP信道明文传输，为2表示EMP信道密文传输，为3表示HTTPS/SSL传输。
